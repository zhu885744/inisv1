# inis CMS 二次开发规范
你希望基于现有项目的 RESTful API 规范，完善二次开发规范文档，以下是结合项目特性、接口调用规则、参数传递等核心内容的完整规范，可直接整合到二次开发文档中：

## 一、API 设计与开发核心规范
### 1.1 RESTful API 基础遵循
- 所有接口必须严格遵循 RESTful 设计原则，请求方法语义明确：
  - `GET`：查询/获取数据（如获取单条/多条数据、统计数量）
  - `POST`：新增数据（如添加单条记录）
  - `PUT`：修改数据（全量更新）
  - `DELETE`：删除数据（软删除/彻底删除）
- 接口路径命名规范：
  - 采用小写英文名词复数形式（如 `/api/articles` 而非 `/api/article`）
  - 多级路径用 `/` 分隔，避免下划线，采用连字符（如 `/api/article-comments`）
  - 禁止在路径中拼接参数（如 `/api/article/123` 合法，`/api/article?id=123` 仅用于复杂查询）

### 1.2 请求头（Content-Type）规范
| 类型                          | 适用场景                          | 开发建议                          |
|-------------------------------|-----------------------------------|-----------------------------------|
| `application/json`（推荐）    | 所有接口默认优先使用              | 支持原生数组/对象传递，开发效率高 |
| `application/x-www-form-urlencoded` | 仅兼容文档演示、简单参数传递场景 | 需按特殊规则传递数组/对象         |

### 1.3 响应结果统一规范
#### 1.3.1 响应结构（所有接口必须遵循）
```json
{
  "code": 200,        // 状态码（仅200为成功）
  "msg": "操作成功",  // 提示信息（用户/开发友好）
  "data": {}          // 数据体（成功时返回业务数据，失败时可为null）
}
```

#### 1.3.2 状态码使用规范（禁止自定义非约定状态码）
| 状态码 | 使用场景                          | 开发注意事项                                  |
|--------|-----------------------------------|-----------------------------------------------|
| 200    | 接口调用成功且数据返回正常        | 所有正常响应必须返回200，data为有效业务数据    |
| 201    | 请求受理但未完成最终处理          | 仅用于异步任务（如批量导入数据、定时任务触发） |
| 204    | 查询成功但无数据                  | data必须为null/空数组，msg注明“暂无数据”       |
| 400    | 参数错误（格式/类型/必填项缺失）  | msg需明确指出错误参数（如“参数article_id不能为空”） |
| 401    | 未登录/Token失效                  | 需引导前端跳转登录页，data返回null             |
| 403    | 权限不足                          | msg需明确权限名称（如“无文章删除权限”）        |
| 404    | 资源不存在（如ID错误、接口路径错）| 区分“资源不存在”和“接口不存在”，msg精准提示    |
| 405    | 接口方法不存在/请求方法错误       | 如POST调用GET接口、路径拼写错误                |
| 429    | 触发QPS限流                       | msg注明“请求过于频繁，请稍后重试”              |
| 500    | 服务器异常（代码错误/数据库异常） | 生产环境msg屏蔽具体错误，日志记录详细异常信息  |

## 二、参数传递规范
### 2.1 通用参数规则
- 所有参数命名采用下划线命名法（如 `create_time` 而非 `createTime`）；
- 必填参数必须在控制器层做非空校验，返回400状态码并明确提示缺失参数；
- 可选参数需设置默认值，避免空指针异常（如分页参数 `page` 默认1，`size` 默认10）。

### 2.2 特殊参数（ids/field）处理规范
#### 2.2.1 ids 参数（主键列表）
- 支持两种传递形式：
  1. 数组形式：`ids: [1,2,3,4]`（推荐，Content-Type=application/json时使用）；
  2. 字符串形式：`ids: "1,2,3,4"` 或 `ids: "1 2a3,4-5&6"`（兼容处理，后端通过正则提取数字）；
- 开发要求：
  - 后端必须统一解析为 `[]int64` 类型后再进行数据库操作；
  - 空ids（空数组/空字符串）需返回400，提示“ids参数不能为空”；
  - 示例解析逻辑：
    ```go
    // 核心解析函数（可封装到facade层）
    func ParseIds(ids any) []int64 {
      var idList []int64
      switch v := ids.(type) {
      case []int64:
        idList = v
      case string:
        // 正则匹配所有数字
        reg := regexp.MustCompile(`\d+`)
        strIds := reg.FindAllString(v, -1)
        for _, s := range strIds {
          id, _ := strconv.ParseInt(s, 10, 64)
          idList = append(idList, id)
        }
      default:
        return []int64{}
      }
      return idList
    }
    ```

#### 2.2.2 field 参数（字段列表）
- 仅支持两种传递形式：
  1. 数组形式：`field: ["id","title","create_time"]`（推荐）；
  2. 字符串形式：`field: "id,title,create_time"`（仅支持英文逗号分隔）；
- 开发要求：
  - 后端必须解析为 `[]string` 类型；
  - 需校验字段合法性（避免SQL注入/查询不存在字段），非法字段直接过滤并提示；
  - 若field为空，默认返回模型所有字段；
  - 示例解析逻辑：
    ```go
    // 核心解析函数
    func ParseField(field any) []string {
      var fieldList []string
      switch v := field.(type) {
      case []string:
        fieldList = v
      case string:
        if v != "" {
          fieldList = strings.Split(v, ",")
        }
      default:
        return []string{}
      }
      // 过滤非法字段（示例：仅允许白名单字段）
      allowFields := map[string]bool{"id":true, "title":true, "create_time":true}
      var validFields []string
      for _, f := range fieldList {
        if allowFields[f] {
          validFields = append(validFields, f)
        }
      }
      return validFields
    }
    ```

### 2.3 数组/对象参数传递规范
#### 2.3.1 Content-Type=application/x-www-form-urlencoded 场景
- 数组：采用 `key[]=val1&key[]=val2` 格式（如 `array[]=1&array[]=2`）；
- 对象：采用 `key[subkey]=val` 格式（如 `json[id]=1&json[age]=20`）；
- 开发建议：前端可使用 `qs` 库自动格式化，避免手动拼接。

#### 2.3.2 Content-Type=application/json 场景（推荐）
- 直接传递原生JSON数组/对象，无需特殊格式化：
  ```json
  {
    "array": [1,2,3],
    "json": {"id":1, "age":20, "nickname":"兔子"}
  }
  ```
- 后端开发要求：
  - 使用Gin的 `ctx.ShouldBindJSON(&struct{})` 绑定参数；
  - 结构体字段需与前端参数名一致，标签示例：
    ```go
    type TestParams struct {
      Array []int               `json:"array"`
      Json  map[string]interface{} `json:"json"`
    }
    ```

## 三、基础接口开发规范
### 3.1 15个基础接口统一实现标准
所有业务模块（文章、评论、用户等）必须实现以下15个基础接口，保证功能一致性：

| 序号 | 接口名称       | 请求方法 | 路径示例          | 核心开发要求                                                                 |
|------|----------------|----------|-------------------|------------------------------------------------------------------------------|
| 1    | 获取指定       | GET      | /api/articles/{id}| 根据ID查询单条数据，支持field参数过滤字段                                    |
| 2    | 获取全部       | GET      | /api/articles     | 支持分页（page/size）、条件筛选、field过滤、排序，返回总数+列表              |
| 3    | 随机获取       | GET      | /api/articles/random | 支持条件筛选、数量限制（limit），返回随机数据列表                            |
| 4    | 查询数量       | GET      | /api/articles/count | 支持条件筛选，仅返回数据总数（data={count:100}）                             |
| 5    | 求和           | GET      | /api/articles/sum  | 指定字段（sum_field）求和，返回总和（data={sum:1000}）                       |
| 6    | 最小值         | GET      | /api/articles/min  | 指定字段（min_field）求最小值，返回结果（data={min:10}）                     |
| 7    | 最大值         | GET      | /api/articles/max  | 指定字段（max_field）求最大值，返回结果（data={max:100}）                    |
| 8    | 查询列         | GET      | /api/articles/list | 与“获取全部”一致，但不带分页，返回完整列表（适合小数据量）                    |
| 9    | 软删除         | DELETE   | /api/articles/soft | 接收ids参数，仅更新delete_time字段（逻辑删除），不物理删除                   |
| 10   | 彻底删除       | DELETE   | /api/articles     | 接收ids参数，物理删除数据，需二次确认（前端）                                |
| 11   | 清空回收站     | DELETE   | /api/articles/recycle | 物理删除所有软删除数据，仅管理员可调用                                       |
| 12   | 恢复数据       | PUT      | /api/articles/recover | 接收ids参数，恢复软删除数据（重置delete_time）                               |
| 13   | 添加数据       | POST     | /api/articles     | 新增单条数据，返回新增记录ID                                                 |
| 14   | 修改数据       | PUT      | /api/articles/{id}| 根据ID更新数据，支持部分字段更新                                             |
| 15   | 保存数据       | POST     | /api/articles/save | 合并添加/修改逻辑：有ID则修改，无ID则新增                                    |

### 3.2 接口测试规范
- 新增接口必须通过项目内置的测试接口（GET/PUT/DEL/POST）验证参数传递正确性；
- 测试重点：数组/对象、ids/field参数的解析是否符合预期；
- 测试通过后，需补充Swagger接口文档，注明参数类型、示例、是否必填。

## 四、开发与调试规范
### 4.1 日志记录规范
- 所有接口必须记录关键日志：
  - 请求日志：包含请求方法、路径、参数、用户ID、IP；
  - 响应日志：包含状态码、耗时、错误信息（500时记录详细堆栈）；
  - 敏感操作日志（删除/修改）：记录操作人、操作内容、操作时间；
- 日志格式统一使用 `facade.Log.Info/Error`，示例：
  ```go
  facade.Log.Info(map[string]any{
    "method": ctx.Request.Method,
    "path": ctx.Request.URL.Path,
    "params": params,
    "user_id": userId,
    "cost": time.Since(start).Milliseconds(),
  }, "接口请求日志")
  ```

### 4.2 异常处理规范
- 禁止直接将原生错误信息返回给前端（如SQL错误、空指针错误）；
- 所有异常必须捕获并转换为约定状态码，示例：
  ```go
  defer func() {
    if err := recover(); err != nil {
      facade.Log.Error(map[string]any{"error": err}, "接口panic异常")
      c.JSON(200, gin.H{
        "code": 500,
        "msg":  "服务器内部异常",
        "data": nil,
      })
    }
  }()
  ```

### 4.3 兼容性规范
- 新增参数必须设置默认值，避免影响旧版本接口调用；
- 修改接口逻辑时，需保留原有响应结构，仅扩展data字段；
- 废弃接口需先标记为“废弃”并保留1个版本周期，再删除。

## 五、安全规范
### 5.1 参数校验
- 所有入参必须做类型/范围校验（如ID必须为正整数、时间格式合法）；
- 禁止直接将用户输入拼接到SQL/Redis命令中，使用Gorm参数绑定/Redis占位符；
- 字段过滤：field参数仅允许查询白名单字段，避免敏感字段泄露。

### 5.2 限流与权限
- 所有接口必须接入QPS限流中间件，默认阈值可配置；
- 权限校验：非公开接口必须验证用户权限，无权限返回403；
- 防刷机制：验证码/短信接口需接入防刷逻辑，避免恶意调用。

### 总结
1. 所有接口严格遵循RESTful规范，响应结构统一为`code/msg/data`，仅200代表成功；
2. 特殊参数`ids`支持任意分隔符的字符串/数组传递（后端正则提取数字），`field`仅支持英文逗号分隔的字符串/数组；
3. 数组/对象参数优先使用`application/json`传递，开发效率更高，`application/x-www-form-urlencoded`需按`key[]/key[subkey]`格式传递；
4. 所有业务模块必须实现15个基础接口，保证功能一致性，同时做好参数校验、日志记录、异常处理和安全防护。
